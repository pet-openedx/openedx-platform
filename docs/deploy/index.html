Deployment

Building a Docker image via GitHub Actions and deploying with Tutor

Overview

The CI pipeline builds a Docker image containing Python dependencies and compiled static assets, then pushes it to the GitHub Container Registry. Tutor pulls this image and handles everything else — collectstatic, gunicorn, nginx, and database initialisation.

Dockerfile — multi-stage build

Three stages keep the final image small: only runtime libraries and pre-built artifacts reach production.

Stage 1
python-build
python:3.11-slim-bookworm
  • apt: build-essential, pkg-config, libmysqlclient-dev, libxmlsec1-dev, libxml2-dev, libxslt-dev
  • pip install base.txt → /install (prefix-isolated)
Stage 2
assets-build
node:20-bookworm
  • apt: python3, python3-pip (for compile_sass.py)
  • npm ci — installs webpack, loaders
  • pip3 install assets.txt — libsass, click
  • npm run build — webpack + compile-sass
Stage 3
production
python:3.11-slim-bookworm
  • apt: runtime libs only (no build-essential)
  • COPY /install from python-build
  • COPY static dirs from assets-build
  • CMD: gunicorn :8000

CI Workflow

File: .github/workflows/docker-image.yml

1

Trigger

workflow_dispatch — manual only. Optional tag input (default: latest).

2

actions/checkout@v4

Full repo checkout on ubuntu-24.04.

3

docker/setup-buildx-action@v3

Enables BuildKit for multi-stage caching and parallel stage execution.

4

docker/login-action@v3 → ghcr.io

Authenticates with GITHUB_TOKEN — no additional secrets needed.

5

docker/build-push-action@v6

Builds and pushes two tags: :<tag> and :<git-sha>. Uses type=gha layer cache — free with GitHub Actions.

Triggering the build

  1. Go to Actions tab in the GitHub repository
  2. Select Docker Image workflow from the left sidebar
  3. Click Run workflow
  4. Optionally enter a tag (e.g. v1.0.0). Leave blank for latest
  5. Click Run workflow

Image tags produced

TagExampleUse
:<input-tag> ghcr.io/pet-openedx/openedx-platform:latest Tutor deployment target
:<git-sha> ghcr.io/pet-openedx/openedx-platform:abc1234… Immutable reference for rollback

Deploying with Tutor

Point Tutor at the new image

tutor config save \
  --set DOCKER_IMAGE_OPENEDX=ghcr.io/pet-openedx/openedx-platform:latest

First-time launch

tutor local launch
Note: tutor local launch runs collectstatic, database migrations, and starts all services (nginx, gunicorn, MySQL, MongoDB, Redis, Celery). This may take several minutes on first run.

Subsequent updates

tutor local stop
tutor local start -d

Rolling back

Use the immutable SHA tag to roll back to a specific build:

tutor config save \
  --set DOCKER_IMAGE_OPENEDX=ghcr.io/pet-openedx/openedx-platform:<git-sha>
tutor local stop
tutor local start -d

Verification

1

Confirm workflow is green

All three stages (python-build, assets-build, production) must complete without errors in the Actions tab.

2

Confirm image is visible in GHCR

Navigate to github.com/pet-openedx/openedx-platform → Packages. The openedx-platform package should appear with both tags.

3

Pull and inspect the image

docker pull ghcr.io/pet-openedx/openedx-platform:latest
docker run --rm ghcr.io/pet-openedx/openedx-platform:latest python -c "import django; print(django.__version__)"
4

Health check after Tutor launch

curl -s http://localhost/heartbeat

Expects {"happy": true} (or HTTP 200).

Repository permissions

The workflow uses GITHUB_TOKEN which is automatically available. Ensure the repository has package write access enabled:

  1. Go to Settings → Actions → General
  2. Under Workflow permissions, select Read and write permissions
  3. Save

For private images, ensure Tutor's host can authenticate to GHCR:

echo <GITHUB_PAT> | docker login ghcr.io -u <github-username> --password-stdin
